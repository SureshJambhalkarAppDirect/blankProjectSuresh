## 1. Overview

### 1.1 Summary

This PRD defines the product requirements for automatically creating or replacing Adobe admin accounts when subscription ownership is transferred in the AppDirect marketplace. When a user performs a "take ownership" (or subscription change) action for an Adobe product subscription, the system must ensure the new owner is provisioned as the Adobe admin—or the existing admin is replaced by the new owner—so that Adobe product administration aligns with marketplace ownership.

### 1.2 Scope

- **In scope:** All Adobe products offered through the AppDirect marketplace where subscription ownership can be transferred.
- **Trigger:** Subscription_Change event (and associated new order) generated by the marketplace when ownership is transferred.
- **Outcome:** New owner has the correct Adobe admin account (created or updated) for the affected subscription(s).

### 1.3 Definitions

| Term | Definition |
|------|-------------|
| **Subscription_Change event** | Marketplace event emitted when a subscription’s ownership is changed (e.g., transfer, take ownership). |
| **Take ownership** | Action by which an admin or another user assumes ownership of an existing subscription from the current owner. |
| **Adobe admin account** | The user identity in Adobe systems that has administrative privileges for the organization’s Adobe products tied to that subscription. |
| **Marketplace** | AppDirect marketplace where subscriptions are created, changed, and managed. |

## 2. Problem Statement

### 2.1 Business Problem

When an employee who owns an Adobe subscription leaves the organization, or when an organization deliberately reassigns subscription ownership to another user, the transfer is reflected in the AppDirect marketplace (new order, Subscription_Change event). However, **Adobe-side administration may still be tied to the previous owner**. As a result:

- The **new owner** cannot perform admin tasks in Adobe (e.g., user management, policy configuration) because they do not have an admin account or the correct role in Adobe.
- The **organization** may retain an admin account for a user who no longer has access to the marketplace, creating security and compliance issues.
- **Support and operations** are burdened by manual steps to align Adobe admins with marketplace ownership.

### 2.2 User Impact

- **New subscription owner (admin):** Cannot administer the Adobe subscription they now own; must rely on support or workarounds.
- **Company administrators:** Risk of orphaned Adobe admin accounts and misalignment between who “owns” the subscription in the marketplace and who is the admin in Adobe.
- **Support teams:** Higher ticket volume and manual intervention to fix admin assignment after ownership changes.

### 2.3 Current State (Gap)

- Marketplace correctly creates a new order and emits a Subscription_Change event when ownership is transferred.
- There is **no automated process** that, upon this event for Adobe products, creates or updates the Adobe admin account to match the new owner.
- Admin alignment, when done at all, is manual and inconsistent.

## 3. Goals & Success Criteria

### 3.1 Goals

1. **Automate admin alignment:** When a Subscription_Change event occurs for an Adobe subscription, automatically create or replace the Adobe admin account with the new subscription owner.
2. **Support all Adobe products:** Behavior applies consistently across all Adobe products in the marketplace.
3. **Single source of truth:** Marketplace ownership is the source of truth; Adobe admin status is derived from it.
4. **Reduce manual work:** Eliminate or minimize manual steps for support and customers when ownership changes.

### 3.2 Success Criteria

| # | Criterion | Measurement |
|---|-----------|-------------|
| 1 | New owner becomes Adobe admin (or existing admin is replaced) for the subscription after Subscription_Change. | Verification in Adobe systems and/or internal admin store. |
| 2 | Process completes for all Adobe products covered by the Subscription_Change event. | No Adobe products in scope are skipped. |
| 3 | Previous owner no longer holds admin for that subscription when replaced. | Audit/API check in Adobe. |
| 4 | Reduction in support tickets related to “admin not set” or “wrong admin” after ownership transfer. | Ticket volume before/after launch. |
| 5 | Event-to-admin-update completes within defined SLA (e.g., &lt; X minutes). | Monitoring of processing time. |

## 4. User Requirements

### 4.1 Primary Users

- **Organization admin (new owner):** Takes ownership of an Adobe subscription in the marketplace and expects to administer it in Adobe without manual steps.
- **Company/IT admin:** Manages subscriptions and expects ownership changes in the marketplace to be reflected in Adobe automatically.
- **Support/Operations:** Expects fewer manual fixes for admin assignment after ownership transfers.

### 4.2 User Requirements (High Level)

| ID | Requirement | Priority |
|----|-------------|----------|
| UR-1 | When an admin user takes ownership of an existing Adobe subscription in the marketplace, the system shall create an Adobe admin account for that user if one does not exist. | Must |
| UR-2 | When an admin user takes ownership of an existing Adobe subscription, the system shall replace the existing Adobe admin with the new owner when an admin already exists for that subscription. | Must |
| UR-3 | The behavior shall apply to all Adobe products that support subscription ownership transfer in the marketplace. | Must |
| UR-4 | The new owner shall be able to access Adobe admin functions (as defined by Adobe) for the subscription after the process completes. | Must |
| UR-5 | The previous owner shall no longer retain admin rights for that subscription once replaced. | Must |
| UR-6 | The system shall rely on the Subscription_Change event (and associated order/user data) as the trigger; no separate manual “sync” step shall be required for normal operation. | Should |
| UR-7 | Failures (e.g., Adobe API errors) shall be detectable and actionable (logging, alerts, optional retry). | Should |

## 5. Solution Overview

### 5.1 High-Level Flow

1. **Trigger:** User performs “take ownership” (or equivalent) in the AppDirect marketplace for an Adobe subscription → marketplace creates a new order and emits a **Subscription_Change** event.
2. **Event consumption:** A backend service (or existing order/event processor) consumes the Subscription_Change event and identifies that the subscription is for an **Adobe product** and that the event represents an **ownership change** (new owner identified).
3. **Identity resolution:** The system resolves the **new owner** (e.g., from event payload or order: user ID, email, marketplace account).
4. **Adobe admin action:**
   - If no admin exists for that subscription/organization in Adobe → **create** an admin account for the new owner.
   - If an admin already exists for that subscription/organization → **replace** that admin with the new owner (deprovision/update previous admin, provision new owner as admin).
5. **Confirmation:** Adobe (and optionally internal store) reflects the new admin. Optional: notify new owner or update UI.

### 5.2 Key Design Decisions (Recommendations)

- **Event-driven:** Use Subscription_Change as the single trigger; avoid polling.
- **Idempotency:** Handle duplicate or retried events without creating duplicate admins or incorrect state.
- **Adobe APIs:** Use Adobe’s documented APIs for admin create/replace (e.g., Admin Console APIs, UMAPI, or product-specific admin APIs as applicable).
- **Scope:** Limit automatic create/replace to Adobe products; other vendors can be addressed in future PRDs.

### 5.3 Components (Conceptual)

- **Event consumer:** Subscribes to Subscription_Change (and possibly order creation) in the marketplace.
- **Adobe adapter/service:** Translates “new owner” + subscription/org context into Adobe admin create/replace calls.
- **Identity mapping:** Maps marketplace user (e.g., AppDirect user ID/email) to Adobe identity (e.g., Adobe ID or enterprise ID) as required by Adobe APIs.
- **Audit/logging:** Records ownership changes and admin create/replace outcomes for support and compliance.

## 6. User Stories

### 6.1 Primary Stories

**US-1 – New owner gets admin when no admin exists**  
As an organization admin who has just taken ownership of an Adobe subscription in the marketplace,  
I want the system to create an Adobe admin account for me automatically  
So that I can manage the Adobe subscription without contacting support.

**US-2 – New owner replaces existing admin**  
As an organization admin who has taken ownership of an Adobe subscription from a departing user,  
I want the previous owner’s Adobe admin account to be replaced with mine  
So that I am the only admin for that subscription and the leaver no longer has admin access.

**US-3 – Consistent behavior across Adobe products**  
As a company administrator,  
I want the same create/replace admin behavior to apply to all Adobe products in the marketplace  
So that I don’t have to remember product-specific processes.

### 6.2 Supporting Stories

**US-4 – Visibility when something fails**  
As an operations or support engineer,  
I want failures in creating or replacing the Adobe admin to be logged and visible (and optionally alerted)  
So that we can fix issues and retry or escalate.

**US-5 – No duplicate admins on retries**  
As the system,  
I want handling of Subscription_Change to be idempotent  
So that duplicate or retried events do not create duplicate admins or inconsistent state.

## 7. Functional Requirements

### 7.1 Event Handling

| ID | Requirement | Acceptance Criteria |
|----|-------------|---------------------|
| FR-1 | The system shall consume Subscription_Change events from the marketplace (or order/event bus). | Event is received and parsed; subscription and new owner are identifiable. |
| FR-2 | The system shall filter for Adobe product subscriptions only. | Non-Adobe subscriptions are ignored for this flow. |
| FR-3 | The system shall extract the new subscription owner (e.g., user ID, email) from the event or linked order. | New owner identity is available for downstream steps. |
| FR-4 | The system shall support idempotent processing of the same event (e.g., by event ID or idempotency key). | Duplicate events do not result in duplicate or conflicting admin updates. |

### 7.2 Adobe Admin Create / Replace

| ID | Requirement | Acceptance Criteria |
|----|-------------|---------------------|
| FR-5 | When the new owner has no Adobe admin for the subscription/organization, the system shall create an Adobe admin account for the new owner. | New owner appears as admin in Adobe for that subscription/org. |
| FR-6 | When an Adobe admin already exists for the subscription/organization, the system shall replace that admin with the new owner. | Previous admin is no longer admin; new owner is admin. |
| FR-7 | The system shall use Adobe’s supported APIs and identity model (e.g., Adobe ID, enterprise ID) for admin create/replace. | Implementation aligns with Adobe documentation and security model. |
| FR-8 | The system shall map marketplace user identity to Adobe identity where required (e.g., email, federated ID). | Correct user is provisioned in Adobe. |

### 7.3 Coverage & Edge Cases

| ID | Requirement | Acceptance Criteria |
|----|-------------|---------------------|
| FR-9 | The behavior shall apply to all Adobe products configured in the marketplace for this flow. | No Adobe product in scope is excluded without explicit configuration. |
| FR-10 | If Adobe API calls fail, the system shall record the failure (log/audit) and optionally support retry or alerting. | Failures are visible and actionable. |
| FR-11 | Optional: Configuration or feature flag to enable/disable this behavior per product or tenant. | Operations can disable or roll out gradually. |

## 9. Assumptions & Dependencies

### 9.1 Assumptions

- Subscription_Change event (or equivalent) is emitted by the marketplace when ownership is transferred and contains (or can be linked to) the new owner and subscription/product details.
- Adobe provides APIs (or existing integration supports) creating and replacing admin users for the products in scope.
- Marketplace user identity (e.g., email, user ID) can be mapped to Adobe identity for admin provisioning.
- “Take ownership” and similar flows result in a single new order and Subscription_Change event per subscription (or well-defined event model).
- Only one “current” admin per subscription/organization context in Adobe is the target state (replace = one in, one out).

### 9.2 Dependencies

- **Marketplace/Order platform:** Reliable emission of Subscription_Change events and order data.
- **Adobe:** API availability, authentication (e.g., OAuth), and documentation for admin create/replace.
- **Identity/SSO (if applicable):** Mapping between marketplace users and Adobe/federated identities.
- **Internal systems:** Any subscription or admin store that must be updated for consistency (if applicable).

---

## 10. Out of Scope

- **Non-Adobe products:** This PRD applies only to Adobe products; other vendors are out of scope unless explicitly extended.
- **Bulk historical migration:** One-time backfill of admins for past ownership changes may be a separate project.
- **Changing Adobe product features:** Any change to what “admin” can do in Adobe is determined by Adobe, not this project.
- **Marketplace UI changes:** Unless required to expose status or errors to the user; focus is backend automation.
- **Multiple admins per subscription:** This PRD assumes a single admin per subscription (replace scenario); multi-admin models are out of scope unless specified later.

## 11. Risks & Mitigations

| Risk | Impact | Likelihood | Mitigation |
|------|--------|-------------|------------|
| Adobe API rate limits or downtime | Delays or failures in admin create/replace | Medium | Retry with backoff; queue; alerting; optional manual fallback. |
| Identity mismatch (marketplace vs Adobe) | Wrong user made admin or failure | Medium | Clear mapping rules; validation; logging of mapped identity. |
| Event payload changes | Parser or logic breaks | Low | Version event contract; integration tests; monitor for schema changes. |
| Replace leaves previous user with partial access | Security/compliance concern | Medium | Use Adobe’s official replace/de-provision flow; verify with Adobe. |
